# Product Proposal - Chronos Agent

## Обоснование идеи

Прикладная задача заключается в автоматизации роли «личного секретаря». Существующие решения (Todoist, Google Calendar) - это пассивные инструменты. Chronos Agent превращает планирование в активный процесс, где система берет на себя когнитивную нагрузку по распределению ресурсов времени.

Агент не просто выполняет одноразовый запрос пользователя, а **поддерживает состояние, периодически анализирует календарь и может инициировать действия самостоятельно**. Архитектура строится на базе **LangGraph**, который управляет состояниями агента и циклом reasoning -> action -> observation.

## Цель и метрики

**Цель:** Снижение временных затрат на менеджмент задач и повышение процента их успешного завершения.

| Тип метрики | Метрика | Целевое значение (PoC) |
| --- | --- | --- |
| **Продуктовая** | Task Completion Rate (TCR) | Рост выполненных задач на 15% |
| **Агентская** | Rescheduling Autonomy | > 70% правок календаря приняты пользователем |
| **Техническая** | Intent Extraction Accuracy | > 85% корректно распознанных дат и сущностей |
| **Техническая** | End-to-end Latency | < 7 секунд (от голоса до ответа бота) |

## Сценарии использования

1. **«На бегу»:** Пользователь записывает голосовую заметку: «Надо сдать отчет до четверга, займусь этим завтра утром». Агент находит завтрашнее утро, проверяет, есть ли свободные слоты, и в соответствии с этим либо ставит задачу в заданный период времени, либо предлагает другие свободные слоты.
2. **«Прокрастинация»:** Пользователь трижды переносит задачу. Агент предлагает разбить задачу на мелкие подзадачи или вовсе отменить её как неактуальную.
3. **«Истечение дедлайна»:** К концу дня пользователь так и не отметил выполненной задачу, запланированную на этот день. Агент автоматически напоминает пользователю об этом, анализирует свободные слоты и предлагает перенести задачу на следующий день/неделю.

## Ограничения и SLO

- **Технические:** p95 latency для Whisper + LLM не более 7 сек. Google Calendar API - не более 100 запросов в минуту.
- **Операционные:** Бюджет на LLM в рамках $10-20 на период разработки. Использование локального Whisper для экономии.

## Архитектура

Проект строится на базе **LangGraph** для управления циклами и состояниями.

**Module A (Perception):** Telegram Bot + Whisper (ASR).  
**Module B (Reasoning):** LLM Agent (ReAct-цикл).  
**Module C (Action):** Google Calendar API.  
**Module D (Memory):** PostgreSQL (хранение стейта LangGraph) + Векторная база (опционально для долгой памяти привычек).



### Основной цикл агента

Chronos Agent работает в итеративном цикле принятия решений (observe -> reason -> act).  
Агент может запускаться тремя типами событий:

1. **User-triggered** - сообщение пользователя в Telegram  
2. **Event-triggered** - изменение состояния календаря или конфликт событий  
3. **Time-triggered** - периодическая проверка задач и напоминаний  

Каждая итерация цикла включает следующие этапы:
#### 1. Perception (получение входных данных)

Система собирает текущее состояние из нескольких источников:

- сообщения пользователя (Telegram: текст или голос)
- текущее состояние календаря
- состояние задач (pending / scheduled / completed)
- системные события (таймеры, cron-триггеры)

Если входное сообщение содержит аудио, оно предварительно обрабатывается через Whisper ASR

---

#### 2. Reasoning (анализ и планирование)

Агент анализирует текущее состояние системы и выполняет:

- извлечение события/задачи пользователя через LLM
- интерпретацию временных ограничений (дедлайн, длительность)
- анализ доступных временных слотов
- формирование гипотезы действия

На этом этапе агент принимает решение какое действие необходимо выполнить.

---

#### 3. Tool Execution (вызов инструментов)

Агент вызывает один из доступных инструментов:

- получить текущие события календаря  
- создать событие  
- перенести существующее событие  
- отправить уведомление  
-  запросить подтверждение пользователя  

Набор инструментов ограничен и валидируется системой.

---

#### 4. Observation (оценка результата)

После выполнения инструмента агент:

- получает результат операции
- обновляет внутреннее состояние (задачи, события, логи)
- оценивает, требуется ли дополнительное действие

---

#### 5. Stop Condition (завершение цикла)

Цикл завершается, если:

- задача успешно запланирована
- пользователю отправлено предложение
- требуется дополнительный ввод пользователя

После завершения агент отправляет пользователю результат или предложение действия.




## Testing & Observability

- **Тесты:** unit‑tests для NLU, интеграционные тесты с календарем, E2E сценарии голос->задача.
- **Метрики и логирование:** latency p95 (ASR+LLM), error_rate, accepted_suggestions_ratio, intent_accuracy.
- **Логирование:** краткие human‑readable объяснения решений + структурированный JSON для Langfuse.
