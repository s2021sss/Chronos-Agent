# Product Proposal - Chronos Agent

## Обоснование идеи

Прикладная задача заключается в автоматизации роли «личного секретаря». Существующие решения (Todoist, Google Calendar) - это пассивные инструменты. Chronos Agent превращает планирование в активный процесс, где система берет на себя когнитивную нагрузку по распределению ресурсов времени.

Агент не просто выполняет одноразовый запрос пользователя, а **поддерживает состояние, периодически анализирует календарь и может инициировать действия самостоятельно**. Архитектура строится на базе **LangGraph**, который управляет состояниями агента и циклом reasoning -> action -> observation.

## Цель и метрики

**Цель:** Снижение временных затрат на менеджмент задач и повышение процента их успешного завершения.

| Тип метрики | Метрика | Целевое значение (PoC) |
| --- | --- | --- |
| **Продуктовая** | Task Completion Rate (TCR) | Рост выполненных задач на 15% |
| **Продуктовая** | Binary Acceptance | > 60% предложений агента подтверждены пользователями |
| **Техническая** | End-to-end Latency | < 7 секунд (от голосового/текстового запроса до ответа) |
| **Техническая** | NLU Success Rate | > 80% запросов с корректно определёнными датой, временем, типом и темой запроса |
| **Оценка качества** |  Calendar State Accuracy (LLM-based) | > 80% соответствия итогового состояния календаря ожидаемому |
| **Надёжность** |  Hallucination Rate | < 10% случаев некорректных или лишних действий |


## Сценарии использования

1. **«На бегу»:** Пользователь записывает голосовую заметку: «Надо сдать отчет до четверга, займусь этим завтра утром». Агент находит завтрашнее утро, проверяет, есть ли свободные слоты, и в соответствии с этим либо ставит задачу в заданный период времени, либо предлагает другие свободные слоты.
2. **«Прокрастинация»:** Пользователь трижды переносит задачу. Агент предлагает разбить задачу на мелкие подзадачи или вовсе отменить её как неактуальную.
3. **«Истечение дедлайна»:** К концу дня пользователь так и не отметил выполненной задачу, запланированную на этот день. Агент автоматически напоминает пользователю об этом, анализирует свободные слоты и предлагает перенести задачу на следующий день/неделю.

## Ограничения и SLO

- **Технические:** p95 latency для Whisper + LLM не более 7 сек. Google Calendar API - не более 100 запросов в минуту.
- **Операционные:** Бюджет на LLM в рамках $10-20 на период разработки. Использование локального Whisper для экономии.

## Архитектура

Проект строится на базе **LangGraph** для управления циклами и состояниями.

**Module A (Perception):** Telegram Bot + Whisper (ASR).  
**Module B (Reasoning):** LLM Agent (ReAct-цикл).  
**Module C (Action):** Google Calendar API.  
**Module D (Memory):** PostgreSQL (хранение стейта LangGraph) + Векторная база (опционально для долгой памяти привычек).



### Основной цикл агента

Chronos Agent работает в итеративном цикле принятия решений (observe -> reason -> act).  
Агент может запускаться тремя типами событий:

1. **User-triggered** - сообщение пользователя в Telegram  
2. **Event-triggered** - изменение состояния календаря или конфликт событий  
3. **Time-triggered** - периодическая проверка задач и напоминаний  

Каждая итерация цикла включает следующие этапы:
#### 1. Perception (получение входных данных)

Система собирает текущее состояние из нескольких источников:

- сообщения пользователя (Telegram: текст или голос)
- текущее состояние календаря
- состояние задач (pending / scheduled / completed)
- системные события (таймеры, cron-триггеры)

Если входное сообщение содержит аудио, оно предварительно обрабатывается через Whisper ASR

---

#### 2. Reasoning (анализ и планирование)

Агент анализирует текущее состояние системы и выполняет:

- извлечение события/задачи пользователя через LLM
- интерпретацию временных ограничений (дедлайн, длительность)
- анализ доступных временных слотов
- формирование гипотезы действия

На этом этапе агент принимает решение какое действие необходимо выполнить.

---

#### 3. Tool Execution (вызов инструментов)

Агент вызывает один из доступных инструментов:

- получить текущие события календаря  
- создать событие  
- перенести существующее событие  
- отправить уведомление  
-  запросить подтверждение пользователя  

Набор инструментов ограничен и валидируется системой.

---

#### 4. Observation (оценка результата)

После выполнения инструмента агент:

- получает результат операции
- обновляет внутреннее состояние (задачи, события, логи)
- оценивает, требуется ли дополнительное действие

---

#### 5. Stop Condition (завершение цикла)

Цикл завершается, если:

- задача успешно запланирована
- пользователю отправлено предложение
- требуется дополнительный ввод пользователя

После завершения агент отправляет пользователю результат или предложение действия.



## Методология оценки

Для оценки качества PoC применяется комплексный подход, совмещающий эталонное тестирование и мониторинг состояния системы в реальном времени.

Качество работы агента оценивается с помощью двух подходов: Unit-testing for NLU (проверка понимания отдельных запросов) и Scenario-based Walkthrough (проверка работы в рамках полного сценария). Алгоритм включает следующие шаги:

- Формируется набор из 20–30 эталонных фраз, покрывающих до 80% типичных запросов: создание задачи с точным временем, перенос на «завтра», добавление задач без времени. Для каждой фразы задаётся ожидаемый JSON-результат, с которым сравнивается ответ агента. Проверяется корректность определения типа события (задача или событие), извлечения даты и времени, а также валидность итогового JSON. В набор также добавляются некорректные запросы (абстрактные фразы или запросы вне зоны ответственности агента), чтобы оценить не только точность, но и склонность агента к галлюцинациям.

- Интеграция с Google Calendar проверяется на том же наборе данных, расширенном для сквозного сценария «от сообщения до записи». На этом этапе агент обрабатывает как текстовые, так и голосовые запросы, а результатом считается не JSON-ответ, а фактическое изменение в календаре. Итоговое состояние проверяется с помощью подхода LLM-as-a-judge на основе размеченных данных. Параллельно замеряются технические показатели, такие как время ответа системы (End-to-end Latency p95).

Отладка реализуется через логирование в Langfuse. Система сохраняет не только структурированные JSON-ответы, но и краткие объяснения логики принятия решений агентом на естественном языке. Это позволяет анализировать причины ошибок в процессе работы LangGraph и отслеживать продуктовую метрику Binary Acceptance - долю предложений агента, которые пользователь подтвердил в интерфейсе (данная метрика рассчитывается на основе данных, получаемых в условиях эксплуатации системы).

## Использование (Use Cases - PoC Phase)

### UC-001: Текстовый ввод → Создание события (базовый путь)
**Сценарий:** "Встреча с Ваней завтра в 14:00 на 1.5 часа"
- **Результат:** Событие создано в Google Calendar, пользователь получил подтверждение

### UC-002: Голосовой ввод → Создание события
**Сценарий:** Пользователь отправляет .ogg файл с речью
- **Результат:** Whisper транскрибирует, затем как UC-001

### UC-003: Конфликт расписания
**Сценарий:** "Встреча завтра в 14:00" но уже есть встреча 14:00-15:00
- **Результат:** Агент предлагает Top альтернативных слота (в порядке приоритета)
- **Пример:** Слот 1: 13:00-14:30 (score 0.85), Слот 2: 15:30-17:00 (score 0.78)

### UC-004: Проактивное переназначение (hourly)
**Сценарий:** Задача "Кодить" с дедлайном сегодня 18:00, но пользователь не отметил выполненной
- **Результат:** В 19:00 агент отправляет: "Задача не выполнена. Перенести на завтра 10:00?" [✅ | ❌]

### UC-005: Низкая уверенность LLM
**Сценарий:** "Мне нужна встреча может завтра может не завтра во какое-то время"
- **Результат:** Агент спрашивает уточнение: "Я не уверен с временем. Уточни пожалуйста"

### UC-006: OAuth истекший
**Сценарий:** Пользователь пытается создать событие, но токен Google expired
- **Результат:** Бот отправляет ссылку переавторизации, процесс возобновляется после авторизации

### UC-007: Таймаут/ошибки LLM (429, 5xx)
**Сценарий:** Mistral API возвращает 429 Too Many Requests
- **Результат:** Retry с exponential backoff (1s, 2s, 4s); если не получится → очередь для повтора позже


---

### Граничные случаи

- **Невалидное время:** "встреча в 25:00" → запросить уточнение
- **Сообщение > 4000 символов** → отклонить
- **Аудио > 20 МБ** → отклонить
- **Таймаут подтверждения > 5 мин** → сессия становится stale, пользователю предложить повторить
